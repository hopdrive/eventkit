The reframe

What changes in the contract

The authoring experience you like stays. One detector, named by the business event. What changes is everything underneath. The handler becomes a declarative job set, and the data contracts, the entry point, and the boundaries between framework, source, and HopDrive domain code all shift with it.

01

The dispatch entrypoint

functions/db-appointments

Before, plugins got re-registered on a global singleton on every invocation. That’s a warm-lambda leak. Now the kit is built once at module scope, register-style.

createEventKit(hasuraEvent) takes the required source as its first arg. kit.use(...) per plugin reads like the old pluginManager.register(...). And the netlifyPlatform adapter (also registered with kit.use) handles payload extraction, the time budget, and the response, so there’s no hand-written getRemainingTimeInMillis line to get wrong. Every dependency is imported.

Nowper-invocation registration
const { listenTo, pluginManager, ObservabilityPlugin,
        TrackingTokenExtractionPlugin } =
  require('@hopdrive/hasura-event-detector');
const { GrafanaLoggerPlugin } = require('../~lib/grafanaLoggerPlugin');
const auth = require('../~lib/functionAuth');
const events = require('./events');

const handler = async (event, context) => {
  if (event?.httpMethod !== 'POST') return { statusCode: 405 };
  if (!auth.hasValidPassphrase(event)) return { statusCode: 401 };
  const { invocationId } = logger.getLabels();

  pluginManager.register(new TrackingTokenExtractionPlugin({…}));
  pluginManager.register(new ObservabilityPlugin({…}));
  pluginManager.register(new GrafanaLoggerPlugin());

  await listenTo(JSON.parse(event.body), {
    invocationId, eventModules: toModules(events) });
  return { statusCode: 200, body: '{"success":true}' };
};
Becomesregister-style, once
import { createEventKit } from '@hopdrive/eventkit';
import { hasuraEvent } from '@hopdrive/eventkit/sources';
import { netlifyPlatform } from '@hopdrive/eventkit/platforms';
import { observability, graphqlSink, batchJobs, loopPrevention, grafanaLogger } from '@hopdrive/eventkit/plugins';
import { getLogger } from '@hopdrive/sdk-server-logger';   // app-side; injected into the bridge
import { batchJobStore } from '../~lib/batchJobStore';   // app-side store.update(id,fields); built-in graphqlBatchJobStore is planned
import auth from '../~lib/functionAuth';
import cfg from '../~lib/config.backend';
import events from './events';

// built ONCE per warm lambda — required source as a param, everything else via use()
const kit = createEventKit(hasuraEvent);                  // required source — first positional arg
kit.use(netlifyPlatform);                                 // optional platform — registered like any plugin
// each I/O plugin takes an INJECTED transport — a sink / store / send seam. The plugin
// stays generic; the destination + secrets arrive as config, never from process.env.
kit.use(observability, { sink: graphqlSink({ endpoint: cfg.graphqlUrl, adminSecret: cfg.graphqlSecret }) });
kit.use(batchJobs,     { store: batchJobStore, logFlush: { intervalMs: 2000 } });
kit.use(loopPrevention, { field: 'updated_by', serviceId: 'event-handlers', codec: { separator: '|', validateCorrelationId: true } });
kit.use(grafanaLogger,  { logger: getLogger() });   // mode 1 — the SDK owns Loki; eventkit just bridges
kit.registerEvents(events);

// kit.handler() owns the Netlify (event, context) signature, the time budget, and the
// response. The `before` hook runs pre-handle; returning a value short-circuits (auth /
// method) before any detection or observability work.
export const handler = kit.handler({
  before: event => {
    // platform-agnostic { status, body } — the adapter's formatRejection shapes it
    // ({ statusCode, body } on classic, a Web Response on v2).
    if (event?.httpMethod !== 'POST') return { status: 405, body: 'Method Not Allowed' };
    if (!auth.hasValidPassphrase(event)) return { status: 401, body: 'Unauthorized!' };
  },
});
02

The detector

events/appointment.ready

We keep the switch over the operation. It’s the readable house style and there’s no reason to change it. Named boolean facts live inside each case, and each case’s return reads like a sentence.

What changes is small. The typed ctx API (ctx.operation / ctx.columnChanged / ctx.newRow) replaces parseHasuraEvent, the row is strongly typed, and MANUAL stays a suppressing case (case 'MANUAL': return false). You branch on ctx.operation with the switch, so there are no inserted()/updated()-style predicate helpers to learn.

Nowhasura-event-detector
const { run, columnHasChanged, parseHasuraEvent } =
  require('@hopdrive/hasura-event-detector');

module.exports.detector = async (eventName, hasuraEvent) => {
  const { dbEvent, operation } = parseHasuraEvent(hasuraEvent);
  switch (operation) {
    case 'INSERT':
      // Spark inserts as 'pending' for address resolution
      return dbEvent?.new?.status === 'ready';
    case 'UPDATE':
      const statusChanged = columnHasChanged('status', dbEvent);
      const isReady = dbEvent?.new?.status === 'ready';
      return statusChanged && isReady;
    case 'DELETE': return false;
    case 'MANUAL': return false;
    default: return false;
  }
};
Becomeseventkit
import { hasuraEvent } from '@hopdrive/eventkit/sources/hasura';
import type { AppointmentRow } from './types';

export const detector = hasuraEvent.detector<AppointmentRow>((ctx) => {
  const { operation, newRow } = ctx;
  switch (operation) {
    case 'INSERT': {
      // Spark inserts as 'pending' first; only fire if it arrives ready.
      const insertedReady = newRow?.status === 'ready';
      return insertedReady;
    }
    case 'UPDATE': {
      const statusChanged = ctx.columnChanged('status');   // helper: stays on ctx
      const isReady       = newRow?.status === 'ready';
      return statusChanged && isReady;
    }
    case 'DELETE':
      return false;
    case 'MANUAL':   // suppress Hasura console edits
      return false;
    default:
      return false;
  }
});
03

The job set, declared not run

events/appointment.ready

There isn’t a handler function anymore. The module declares its jobs as a static array and the runtime runs them (ADR-025), so there’s no run() call and no function body where we branch through the work.

The shared setup that used to live in the handler, mostly the sdk.apollo.initialize(...) pattern that got threaded into almost every job, now becomes prepare(ctx): it runs once and its result is merged into each job’s ctx.input automatically, and from there a job can layer on its own input as a static object or a pure (ctx) => ... mapper. Run options like parallel and continueOnFailure move up onto the module too.

There’s no resolve() here, on purpose. Hasura reads the status code, not the body, so the platform adapter answers it for us: a foreground function returns 200 once the run is accepted, a background one returns 202 straight away. A failing job doesn’t flip that to a non-2xx, because a 500 would make Hasura replay the whole event and re-fire the siblings that already succeeded. Retrying one job is BatchJobs’ concern, not Hasura’s. resolve only comes out when the caller actually keys on the response, like a vendor webhook.

The result is a declarative record. An engineer, or the flow tooling, can read it top to bottom and understand what it does without having to mentally execute a handler.

Nowhandler runs the jobs
const { run, parseHasuraEvent } = require('@hopdrive/hasura-event-detector');
const { job } = require('../../~lib/scoped-job');
const { sendOfferSMS, sendAppointmentOfferedEmailToOrg } = require('../jobs');
const config = require('../../~lib/config.backend');

module.exports.handler = async (eventName, hasuraEvent) => {
  const { dbEvent } = parseHasuraEvent(hasuraEvent);
  const sdk = require('@hopdrive/sdk');

  // shared setup, threaded into every job by hand ↓ (this is the `prepare` case)
  sdk.apollo.initialize({ admin_secret: config.graphqlSecret, … });

  let jobs = [];
  jobs.push(job(sendOfferSMS,                 { sdk, appointment: dbEvent?.new }));
  jobs.push(job(sendAppointmentOfferedEmailToOrg, { sdk, appointment: dbEvent?.new }));
  return await run(eventName, hasuraEvent, jobs);   // handler executes them
};
BecomesdefineEvent — declarative
import { defineEvent, job } from '@hopdrive/eventkit';
import { detector } from './appointment.ready.detector';   // the switch from §01
import { initSdk } from '../~lib/sdk';
import { sendOfferSMS, sendAppointmentOfferedEmailToOrg } from './jobs';
import type { AppointmentRow } from './types';

export default defineEvent<AppointmentRow>({
  name: 'appointment.ready',
  detector,
  // runs ONCE; its result is merged into every job's ctx.input
  prepare: (ctx) => ({ sdk: initSdk(), appointment: ctx.newRow }),
  // a STATIC list, the runtime runs them. no run() call, no body to branch in.
  // a bare function reference IS a job; only wrap with job() to pass options.
  jobs: [
    sendOfferSMS,                            // bare fn, inherits { sdk, appointment }
    sendAppointmentOfferedEmailToOrg,
  ],
  run: { mode: 'parallel', continueOnFailure: true },   // optional — these are the defaults
});

Tip: a job in the array is just a function reference. You only wrap it with job(fn, { ... }) when you need to override settings for that one job, like retries, timeoutMs, metadata, or per-job input. Pass those as the options object, the second argument to job().

The rule we’re cementing: no conditional jobs

jobs is a static array and the runtime runs it. There’s no run() call and no handler body, so there’s nowhere to put an if that adds or skips a job. The job set is fixed by construction (ADR-025), not by convention. The branded job() and a register-time throw are just a backstop.

This is the point. A conditional push buried in a handler is invisible. Observability can’t record a job that may or may not have been added, and the react-flow view and the Flow Manifests can’t draw an edge they can’t see. The job set stops being statically knowable, which is exactly what those layers need from it. A static array stays legible to every tool that reads it.

Nowhandler pushes conditionally
const jobs = [sendReceipt];
if (wasRefunded) jobs.push(sendRefundNotice);
return run(eventName, event, jobs);
Becomesstatic array
jobs: [sendReceipt, sendRefundNotice],
// sendRefundNotice no-ops when nothing to refund

A real condition still has a home. Either it decides which business event occurred, so it belongs in a detector with a name of its own, or it just means one job has nothing to do, so that job returns early on its own ctx.input. Neither one touches the array.

⚙

Controlling how the jobs run

module run: {} + job(fn, opts?)

Two levels, both declarative. The module sets the batch policy in run: {}. Each job can add or override its own in job(fn, { ... }). The defaults match the old Promise.allSettled fan-out, but now they’re a pinned contract (ADR-014), not implicit behavior. mode defaults to parallel and continueOnFailure to true, so a flaky job can never block a sibling on the money path. These only control how jobs run. They can never change which jobs run (ADR-025).

Module · run: { … }

  • mode — 'parallel' (default) or 'series'. Parallel launches all jobs at once; series awaits each before the next.
  • continueOnFailure — default true. A job failure never stops its siblings; run returns a complete JobExecution[] of successes and failures. In series with this false, the first failure stops the rest, which are marked skipped.
  • timeoutMs — a batch-level budget across the whole set.

Per-job · job(fn, { … })

  • retries — in-process retry attempts core makes (durable, delayed retries are BatchJobs’ job, not this).
  • timeoutMs — per-job deadline; the runtime races the job and marks it timed_out.
  • continueOnFailure — per-job override of the module default.
  • name — stable job identity for observability / flow (else fn.name; set it explicitly so a bundler can’t mangle it).
  • tags — labels for filtering & grouping in the trace.
  • input — live request-scoped data (a static object or a pure (ctx) => … mapper); merges highest, over prepare and plugin baselines.
  • metadata — serializable annotations, persisted by BatchJobs and recorded by Observability.

Of these, only retries, timeoutMs, and continueOnFailure govern how a job runs; name/tags/input/metadata are identity and data. RunOptions live on the module’s run (ADR-025) rather than a run() call — the runtime applies them when it executes the declared set.

The job function

db-appointments/jobs/sendOfferSMS

A job stops being a three-argument function digging through a magic options bag. It gets one typed context. Live dependencies come from ctx.input, and its identity comes from ctx.job.

Logging changes too. Today a job imports getLogger() at module scope and the scoped-job wrapper bolts correlation onto it. Under EventKit the job logs through ctx.log, which the runtime has already scoped with invocation, correlation, and job ids, and fanned out to whatever logging plugins are registered (Grafana, observability, the batch row).

The job is plugin-agnostic. Same input from any caller, same result.

Source-agnostic, made concrete

An inbound webhook is just another event

Vendor webhooks (Acertus, Stripe, Spark) are bespoke functions today. Hand-rolled signature checks, ad-hoc try/catch, no correlation, no retry, no trace. Point a webhook() source adapter at the same endpoint and the request becomes an EventEnvelope. From there the module shape, the jobs, the observability, and the durability are the same as a database event. What’s webhook-specific is the detector context (signature, vendor, event type), and a resolve when the vendor cares about the HTTP response it gets back.

↪

A vendor webhook event

acertus.order.created

The business event keeps a business-semantic name, acertus.order.created, not an HTTP route. Detection becomes explicit and observable.

Two things are webhook-specific. The detector reads signatureVerified / vendor / eventType instead of DB rows. And because Acertus keys its retries on the HTTP status, the module adds a resolve that owns the response. resolve returns a value for a 200 ack, or throws ClientError(status) for a specific status (a 4xx tells Acertus not to retry a bad payload). The real work stays a durable job, retried on its own, so a fast ack doesn’t cost you durability.

Nowbespoke function
// functions/webhook-acertus — outside the framework
export const handler = async (event) => {
  const body = JSON.parse(event.body);
  if (!verifyAcertusSignature(event.headers, body))
    return { statusCode: 401 };
  if (body.type !== 'order.created') return { statusCode: 200 };

  try {
    await createOrderFromVendor(body, sdk);   // no record
    await publishEventLog('acertus.order.created', body);
  } catch (e) {
    console.error(e);   // no correlation, no retry, no trace
    return { statusCode: 500 };
  }
  return { statusCode: 200 };
};
Becomeswebhook source adapter
// events/acertus.order.created.ts
import { defineEvent, job, ClientError } from '@hopdrive/eventkit';
import { webhook } from '@hopdrive/eventkit/sources';
import { publishEventLog, createOrderFromVendor } from './jobs';
import type { AcertusOrder } from './types';

export const detector = webhook.detector((ctx) => {
  const { signatureVerified, vendor, eventType } = ctx;
  const isAcertus    = vendor === 'acertus';
  const orderCreated = eventType === 'order.created';
  return signatureVerified && isAcertus && orderCreated;   // forged requests never fire
});

export default defineEvent<AcertusOrder>({
  name: 'acertus.order.created', detector,
  prepare: (ctx) => ({ order: ctx.body }),
  // resolve owns the response Acertus keys on for retries.
  // return a value → 200 ack. throw ClientError(status) → that status.
  resolve: (ctx) => {
    if (!ctx.prepared.order?.dealershipId)
      throw new ClientError(400, 'missing dealership');   // 4xx, Acertus won't retry a bad payload
    return { received: true };                          // 200, fast ack
  },
  // the real work is a DURABLE job, retried on its own, not by the vendor
  jobs: [
    job(createOrderFromVendor, { retries: 3 }),
    job(publishEventLog,       { metadata: { vendor: 'acertus' } }),
  ],
});

Signature verification moves into the adapter. It runs before normalize() and surfaces ctx.signatureVerified, so a forged request never fires the event. The endpoint is the same register-style shape as a db-* function, except the platform adapter is what maps the module’s resolve (or a thrown ClientError) onto the HTTP response:

import { createEventKit } from '@hopdrive/eventkit';
import { webhook } from '@hopdrive/eventkit/sources';
import { netlifyV2Platform } from '@hopdrive/eventkit/platforms';
import { observability, graphqlSink } from '@hopdrive/eventkit/plugins';
import { acertusSignature } from './security';
import events from './events';

const kit = createEventKit(webhook, { vendor: 'acertus', verify: acertusSignature })   // source + its config
  .use(netlifyV2Platform)   // maps resolve → 200, ClientError → its status, a crash → 500
  .use(observability, { sink: graphqlSink({ endpoint: cfg.graphqlUrl, adminSecret: cfg.graphqlSecret }) })
  .registerEvents(events);

export default kit.handler();   // the platform owns the Response

When does work go in resolve vs a job? If the vendor’s response has to reflect the outcome of the work, do that work in resolve and return or throw based on it. resolve never sees job results (ADR-025), so it can’t wait on a job. If a fast ack plus durable processing is fine, keep the work in jobs and let resolve just validate and ack. That’s usually the better trade: the vendor gets a quick 200, and your own durable retries handle the processing instead of leaning on the vendor to retry. (The three response shapes the platform can produce are laid out in the Platform adapters section.)

The payoff
Every vendor webhook now produces the same Invocation → Event → Job records, correlation IDs, buffered observability, retries, durable jobs, and Flow/Compare as a database event — for free, by adopting the adapter. The integration stops being a snowflake.

The plugin model

Infrastructure that the core knows nothing about

EventKit core normalizes, detects, and runs jobs. Everything else is a plugin. Observability, durability, loop prevention, logging, alerting, all of it. A plugin is an independent module that depends only on EventKit’s public types, implements the lifecycle hooks it cares about, and is registered explicitly with kit.use(...). The runtime works fine with zero plugins.

Entirely independent from core

  • A plugin imports only public types from @hopdrive/eventkit — never internals (scheduler, dispatch, retry loop).
  • Core has zero knowledge of any specific plugin; it just calls hooks in registration order.
  • Plugins are versioned and tested independently — all five prebuilt ones ship in the eventkit repo (/plugins/*); a HopDrive or third-party package is only needed for an SDK-coupled extra.
  • They self-correlate from the IDs already on the context (invocation, envelope, job execution) and never require a job or handler to pass them an object — so jobs stay plugin-agnostic. A job runs identically whether or not a plugin is loaded.

The whole API they consume

  • Declaration fields (not hooks): name, provides? / requires? — the singleton role a plugin fills / depends on (e.g. requires: ['source:hasura']), validated at onInit.
  • Notification hooks (void): onInit, onInvocationStart/End, onJobStart/Progress/Checkpoint/Log/End, onError, onFlush, onShutdown.
  • Transform & capability hooks (return a value): configureInvocation, augmentJobContext, normalize, …
  • Typed contexts passed to each hook: InvocationContext, JobContext, JobExecution, EventEnvelope, ErrorContext.
  • Core utilities: serializeError, serializeOutput, replaceCircularReferences.

How plugins compose — three hook shapes

The rule that holds it together: the built-in default for every step always runs — plugins layer around it, they never subclass or override it. “Default behavior plus a little extra” is the common case, not replacement. A hook is exactly one of three shapes, and its name tells you which: on… returns nothing (observe); a bare verb returns a value (the runtime wants something back).

1 · Notification on… · void

Observe; can’t change a value. Start/End for spans, Before/After to do a little extra around a built-in step. N plugins all fire.

2 · Delta transform verb · returns partial

Contribute to a value the runtime assembles — configureInvocation, augmentEnvelope, augmentJobContext. Default always contributes; you only add. “Default + extra” is the only mode.

3 · Singleton capability verb · one owner

normalize, extractPayload, … exactly one provider (provides). The only replaceable shape — and to reuse the default you get it injected as base (DI), never extends/super.

So a user almost never replaces anything: for a little extra they reach for an onAfter… notification (no value change) or a delta transform (to influence the value), and the built-in keeps running underneath. Replacement-with-injected-default is the rare escape hatch for the one-owner capabilities.

When hooks fire · one invocation, top to bottom

  1. build · once onInit
  2. inbound · platform + source extractPayload · buildRequest onBeforeNormalize normalize augmentEnvelope · configureInvocation onAfterNormalize
  3. invocation onInvocationStart

What a plugin looks like to implement

a 20-line alerting plugin

A plugin is a factory that returns the hooks it implements. This one forwards failures to Slack. Obviously not core business, obviously independent. Notice it imports types only, and its secret arrives as injected config, never from process.env.

plugins/slack-alerts.ts
import type { EventKitPlugin, ErrorContext } from '@hopdrive/eventkit';

// a plugin is just a factory returning the hooks it cares about
export function slackAlerts(config: { webhookUrl: string }): EventKitPlugin {
  return {
    name: 'slack-alerts',
    // implement ONLY the hooks you need — the rest are skipped
    async onError(ctx: ErrorContext) {
      const { phase, jobName, eventName, error, correlationId } = ctx;
      await fetch(config.webhookUrl, {
        method: 'POST',
        body: JSON.stringify({
          text: `🚨 ${phase} failed in ${jobName ?? eventName}`
              + `: ${error.message} (corr ${correlationId})`,
        }),
      });
    },
  };
}

How they’re loaded

explicit registration, once

No auto-discovery. It’s hostile to serverless bundlers. The kit is built once at module scope with its single source, and plugins are added register-style with kit.use(plugin, config?). You pass the plugin itself (not a call) plus optional config, and the kit instantiates it. One legible line each, reading like the old pluginManager.register(...). The runtime calls each plugin’s matching hooks at every lifecycle point.

functions/db-appointments
const kit = createEventKit(hasuraEvent);

kit.use(netlifyPlatform);
kit.use(observability, { sink: graphqlSink({ endpoint: cfg.graphqlUrl, adminSecret: cfg.graphqlSecret }) });   // ← injected transport (sink)

kit.use(slackAlerts, { send: postToSlack(cfg.slackWebhookUrl) });   // ← pass the plugin + an injected send seam; app holds the secret

Secrets & environment dependencies

The rule that keeps plugins portable and testable: a plugin never reaches into process.env as its contract. It declares its config shape — including secrets — and the application resolves and injects them. Secrets are framework-handled in one place, not scattered across plugin internals.

The app resolves secrets once, at module load. config.backend reads process.env (Netlify env vars, or a secrets manager) a single time per cold start.
Injected into plugin factories at createEventKit(). Because the kit is module-scoped, secrets resolve once per cold start and are reused across warm invocations — no per-request env reads.
Validated at init — fail fast. createEventKit() validates required plugin config before the first invocation, so a missing graphql.adminSecret throws clearly at boot, not mid-event in production.
Secrets live in plugin config only. They never enter ctx.input, metadata, or batch_jobs — so they are never persisted, logged, or written to the observability tables. (This is exactly why the live/persisted channel split matters.)
Per-request overrides & rotation flow through configureInvocation / RequestContext.pluginConfig when a value must change per call — still never read from the environment by the plugin itself.

Source adapters

The time-budget question, generalized

Every runtime falls into one of three buckets — the adapter collapses all three into the single RequestContext.getRemainingTimeMs, so cancellation and flush-before-kill work identically everywhere:

A · Native countdown

Lambda, Netlify classic. Forward context.getRemainingTimeInMillis(). Precise cancel + flush.

B · Computed deadline

Vercel, GCP, Azure, Cloudflare, Netlify v2. Adapter computes maxMs − elapsed from a configured max.

C · No deadline

Long-running servers, local/test. Budget is unbounded; rely on per-job timeoutMs + graceful shutdown.

The prebuilt plugins

Five plugins ship with EventKit. Each card shows what it does, the core hooks and API it leans on (tagged by composition shape above), and what you get for free by registering it. The Was → line records how each one changes from today’s hasura-event-detector.

Where each one lives follows one rule (ADR-024). If a plugin’s behavior is fully determined by injected config, it ships built-in inside @hopdrive/eventkit. All five do, loop-prevention included (the updated_by field is config, not a hardcoded convention). A plugin only leaves core if it imports a HopDrive SDK, knows a domain table, or hardcodes a cross-service agreement. HopDrive’s leftover layer is just config presets, not logic.

This is what keeps the I/O plugins generic: each one owns its logic (observability owns the record hierarchy + buffering, batchJobs owns the durable lifecycle, the transports own log/error shaping) but takes its destination as an injected seam — a sink for observability, a store for batchJobs, an injected logger (or direct Loki) for grafana, a send for sentry. The plugin never knows whether it’s talking to Hasura, an HTTP collector, the sdk-server-logger queue, or a test spy; the destination + its secrets arrive as config, and no plugin imports a HopDrive SDK — the app injects it (e.g. getLogger() into the grafana bridge). EventKit ships a built-in graphqlSink (and a graphqlBatchJobStore is planned) so HopDrive wires them in one line, while staying trivially testable and swappable.

…and the inbound flow that structure produces:

Now · header-routed monolith

  1. Spark POSTs the webhook to v1-handleSparkWebhook (a Netlify v2 function).
  2. The dispatcher verifies the signature, reads the event HTTP header, JSON-parses the body.
  3. It looks up sparkEventHandlers[eventType] — a hand-built map keyed by the header. This map is the entire “event system”: routing is the only detection.
  4. handleAppointmentCreated does everything: validate → resolve dealership → sync config → dedup → map → appointments.create → return { success, httpStatus }.
  5. The dispatcher maps httpStatus → the Response Spark sees; Sentry is fired ad hoc (some in the handler, some in the dispatcher catch). No invocation/job records, no retries, no durability.

Becomes · detector → declarative jobs

  1. Spark POSTs → v1-handleSparkWebhook; kit.handler() runs.
  2. The webhook source adapter verifies the signature and normalizes the Request → EventEnvelope (vendor:'spark', eventType from the header, body, signatureVerified).
  3. Each module’s detector runs; spark.appointment.created matches vendor==='spark' && eventType==='appointment.created'. Routing is now explicit, observable detection with a business event name.
  4. Its module declares a static jobs set the runtime runs (upsertAppointmentFromSpark, syncSchedulerConfig) — the same payloadMapper/dealershipMapping helpers, now called from a job.
  5. Each job is recorded (Invocation→Event→Job), retryable, durable-capable; errors route to the sentry() plugin. The netlifyV2Platform adapter maps the result → the HTTP Response, preserving Spark’s status contract.

  if (await appointments.getByExternalId({ source: 'spark', externalId: payload.externalId }))
    return { outcome: 'idempotent' };                  // → 200
  return appointments.create(mapSparkPayloadToCreateInput(payload, dealership));

Why it’s a bigger lift than db-*

A

There’s no detector/handler split to port

The db-* repos already separate detection from handling. Here, routing-by-header is the detection and the handler is a monolith. You must introduce a detection layer and decompose each handler into jobs — real refactoring, not a mechanical rename.

Refactor
B

The vendor HTTP-status contract is load-bearing

Vendors retry based on the status you return — and conventions like “200 on ImmutableFieldError (don’t retry)” or “400 on validation” are semantic. The entrypoint must map InvocationResult → the exact status the vendor expects. Get it wrong and you cause vendor retry storms or silently dropped events — the reverse-integration analog of the money-path risk.

Status mapping = shadow-test it
C

Web Request, not the classic lambda event

These run on Deno/Netlify with the Web Request/Response API (and read a header for the event type). The webhook() adapter must normalize a Request, not event.body — a distinct entry shape from the db-* functions.

Adapter work
D

It’s N repos, so fix the template once

Spark, Super Dispatch, Central Dispatch, CarsArrive, Plateau all share the pattern — and there’s an external-integration-template. Migrate the template to EventKit first; every new integration then gets observability, durability, and Flow/Compare by default, and existing repos follow the same recipe.

Leverage
The win
Once migrated, every integration — inbound and outbound — speaks the same event language and lands in the same observability and Compare tooling as the core platform. A Spark appointment sync and an appointment.ready database event become equally legible, traceable, and replayable. The integrations stop being a parallel universe.

What the records unlock

Expected vs. actual, for the whole platform

Every source now produces the same structured records, whether it’s a database event, a webhook, a cron tick, or a reverse integration. So one piece of tooling can reason across all of them. This is the strategic endpoint, and it’s deliberately phased behind the runtime migration.

◆ Flow Manifests — phased

Declare a business process — its expected events, jobs, and branches — as a source-controlled contract. The manifest owns meaning; generated metadata verifies structure. Validated in CI.

◆ Compare Mode — phased

Overlay observed runtime onto the expected flow and classify the differences — missing steps, unexpected side effects, retries, timeouts. Proven on one high-value flow before generalizing; never a production gate.

The one-line summary

The detector layer is a port; the declarative jobs + prepare → job data contract is the migration. Get the input/metadata split, the typed context, the per-invocation entry, and the loop-prevention contract right, and the other 240-odd modules follow the same mechanical pattern.

Entry point

Build one kit per warm lambda at module scope; everything else hangs off it.

createEventKit(source, config?): EventKit

Construct a kit with its one required source (the first positional arg — ADR-019). config is the source’s own config (e.g. createEventKit(webhook, { verify })).

kit.use(plugin, config?): EventKit

Register a plugin/factory (the value itself, not a call) plus optional config; the kit instantiates it lazily. Chainable. The platform adapter registers here too.

kit.registerEvent(module): EventKit kit.registerEvents(modules[] | record): EventKit

Register one or many defineEvent modules. Explicit only — no filesystem auto-discovery. Throws on a duplicate event name or a non-job entry in any module’s jobs.

kit.handler(opts?: { before? }): (...args) => response

Zero-boilerplate entry: the registered platform adapter owns the runtime signature and the response shape. before(...args) runs pre-dispatch; returning a value short-circuits (auth / method) and the adapter’s formatRejection shapes it ({ status, body } → platform-native).

kit.handle(rawPayloadOrArgs, request?): Promise<InvocationResult>

Manual entry — forward raw platform args; the adapter extracts payload + time budget. Use when you need to wrap dispatch yourself.

kit.validate(): void kit.shutdown(): Promise<void>

validate() runs plugin/capability resolution + module checks (also run on first handle). shutdown() flushes and tears down plugins.

Event module — defineEvent

A declarative record (ADR-025): the runtime runs the declared jobs. There is no handler function.

defineEvent<TPayload>({ name, detector, prepare?, jobs?, resolve?, run?, metadata? }): EventModule
  • name — the business event name (the module’s identity; preserved verbatim across migration).
  • detector — (ctx) => boolean | Promise<boolean>. The predicate; house style is switch (ctx.operation) with named booleans.
  • prepare? — (ctx) => shared. Runs once before the jobs; its result merges into every job’s ctx.input (shared sdk, fetched rows, closures).
  • jobs? — JobDefinition[], a static array literal of job(fn, opts?). The runtime runs them; no conditional inclusion, no fan-out, no inter-job deps. Optional only when resolve is present — a module must declare jobs and/or resolve.
  • resolve? shipped — (ctx) => output, the request/response seam (ADR-026): computes the invocation’s response value for a request/response source (hasuraAction, a status-contract webhook). Source-agnostic; the source’s platform adapter maps the return → the wire. It MUST NOT read job results — jobs run alongside as sibling-ignorant side effects.
  • run? — RunOptions for the batch (mode / continueOnFailure / timeoutMs).
  • metadata? — registration-time hints for tooling (description, tags, owner, flowHints); nothing depends on it at runtime.

Jobs — job(), JobOptions, RunOptions

A job is (ctx) => result. Authored via job(fn, opts?); see “Controlling how the jobs run” above for the option semantics.

job<TInput, TResult>(fn, options?): JobDefinition

Wrap a job function into a branded definition. Options:

  • retries? number · timeoutMs? number · continueOnFailure? boolean — execution control (per-job override of the module default).
  • name? string — stable identity for observability/flow (else fn.name; set explicitly to survive minification).
  • tags? string[] · metadata? Record — labels & serializable annotations (persisted + traced).
  • input? TInput | (ctx: JobInputContext) => TInput — live request-scoped data (object or pure mapper); never persisted; merges highest over prepare + plugin baselines.
RunOptions = { mode?: 'parallel'|'series', continueOnFailure?, timeoutMs?, metadata? }

Pinned defaults (ADR-014): mode:'parallel', continueOnFailure:true. Supplied on the module’s run, not a call.

Sources

A source normalizes an inbound signal into an EventEnvelope and supplies the typed detector/job context. Exactly one per kit. Two classes (ADR-026): fire-and-forget event sources (detect → jobs; response is an ack) and request/response sources (detect → resolve a payload + optional side-effect jobs).

hasuraEvent shipped

Hasura DB event triggers (sourceType:'database'). Authoring helpers hasuraEvent.detector<Row>(fn) / hasuraEvent.prepare<Row>(fn) carry types. Detector context: operation (branch on it with a `switch`; `case 'MANUAL': return false` suppresses console edits), oldRow/newRow/row, columnChanged()/columnAdded()/columnRemoved(), previousValue()/currentValue().

hasuraCron shipped

Hasura scheduled (cron) triggers (sourceType:'cron'). Context: scheduleName, scheduledAt, payload.

webhook(config) shipped

Vendor webhooks ({ vendor, verify, eventTypeHeader }); context exposes signatureVerified, vendor, eventType, body. Covers event-handlers’ webhook-* (Stripe, Twilio, Dealerware) and the reverse-integration repos. Mostly fire-and-forget; a status-contract vendor (Stripe) uses resolve + a thrown ClientError(status,…) that the platform maps to the HTTP status.

hasuraAction shipped

A request/response source (sourceType:'action') for Hasura Actions — custom GraphQL business logic gated by Hasura’s permission model. Hasura POSTs { action:{name}, input, session_variables, request_query }; the module’s resolve returns the output type. Context: actionName, input, sessionVariables (role/userId/email), requestQuery?. The response is mapped by the generic platform adapter you register (netlifyV2Platform/netlifyPlatform/lambdaPlatform) — there is no dedicated action platform: resolve’s return → 2xx + the output JSON, a thrown ActionError(message, code?) → 4xx + { message, extensions:{ code? } }. Bespoke app-* endpoints are converted to actions over time (security via RBAC; downstream effects traced by correlation id), not migrated as HTTP — see the migration playbook.

ClientError(status, message) · ActionError(message, code?, extensions?)

The two ways a resolve (or a job) reports failure — kept separate on purpose, because the two callers react to different things. ClientError carries an HTTP status; the platform responds with that exact status code — use it where the caller keys on the status (a status-contract webhook like Stripe deciding whether to retry). ActionError carries a GraphQL error code (+ optional extensions), not a status; the platform responds HTTP 4xx + { message, extensions: { code? } } — the shape Hasura turns into a GraphQL errors[] entry the client reads. The HTTP status is invisible to that client, so there’s nothing meaningful to choose; what matters is the machine-readable code. One speaks the HTTP-status world, the other the GraphQL/Hasura world — folding them into one type would force a webhook author to invent a code and an action author to invent a status the client never sees.

Plugins

Generic, config-driven, registered via kit.use. I/O plugins take an injected transport seam (sink/store/logger/send) so they never read process.env.

observability({ sink, strict? }) shipped

Buffers the Invocation→Event→Job record hierarchy; flushes once per invocation through sink(batch). Best-effort unless strict.

graphqlSink({ endpoint, headers? | adminSecret? }) shipped

Built-in observability sink that bulk-upserts to the canonical observability tables over Hasura GraphQL.

batchJobs({ store, logFlush? }) shipped

Registration-emergent durability; requires:['source:hasura']. Injects the batch_jobs row’s input, persists lifecycle via the injected store.update(id, fields). A built-in graphqlBatchJobStore is planned — provide your own store meanwhile.

loopPrevention({ field?, serviceId?, codec?, read? }) shipped

Reads the inbound provenance field (HopDrive: updated_by) into envelope.meta; outbound ctx.trackingToken is a core seam. Also exports createTokenCodec.

grafanaLogger({ logger | grafana, source?, minLevel?, flush? }) shipped

Log bridge. Mode 1: inject a logger (getLogger() from sdk-server-logger). Mode 2: grafana:{ endpoint, auth, labels } for direct Loki.

sentry({ dsn?, endpoint?, environment?, release?, send? }) shipped

Forwards onError to Sentry. Inject send to delegate to the official SDK (the default sender is a placeholder).

Platform adapters

Map a deployment runtime’s invocation signature, time budget, and response shape. Register via kit.use(...); the source stays the positional arg.

  • netlifyPlatform() shipped — Netlify classic (event/context → { statusCode, body }; native time budget).
  • netlifyV2Platform({ maxExecutionMs? }) shipped — Netlify v2 / Web Request→Response (the reverse-integration repos).
  • netlifyBackgroundPlatform({ maxExecutionMs? }) shipped — 15-min background / 202 (the live batch-jobs watch view).
  • lambdaPlatform() shipped — AWS Lambda (getRemainingTimeInMillis budget).

No dedicated action platform. The request/response contract composes onto these generic adapters: any of them maps a module’s resolve output → 2xx + the output JSON; a thrown ClientError(status, …) → that exact HTTP status (status-contract webhooks, e.g. Stripe); a thrown ActionError(message, code?) → 4xx + { message, extensions:{ code? } } (the Hasura GraphQL-error shape). So a Hasura Action is just createEventKit(hasuraAction).use(netlifyV2Platform).

Core types

The runtime vocabulary. All exported from the root / @hopdrive/eventkit/core.

EventEnvelope

Normalized inbound signal: id, source, sourceType, receivedAt, correlationId, payload, meta, raw?.

DetectedEvent

A fired business event: name, ids, detectedAt, detectorDurationMs, envelope.

DetectorContext / HandlerContext / JobInputContext

Detection-only / post-detection / the context a per-job input mapper sees (handler ctx + prepared).

JobContext

What a job receives: input, event, envelope, job (id/name/attempt/metadata), trackingToken, log, progress(), checkpoint(), signal?.

JobExecution / JobExecutionStatus

One job attempt’s result. Status: queued|running|completed|failed|skipped|timed_out|cancelled.

InvocationResult / EventOutcome

The aggregate of one invocation: ok, events[] (each with error?), durationMs, timedOut?.

RequestContext / InvocationContext / KitContext

Per-request input / per-invocation context threaded to plugins / kit-level context injected at onInit.

LogEntry / LogLevel + tiered loggers

DetectorLogger / HandlerLogger / JobLogger; entries carry correlation ids + scope.

SerializedError / ErrorContext / Capability

Safe error shape; the onError payload; the provides/requires token ('source:hasura').

EventKitPlugin / SourceAdapter / PlatformAdapter

The unifying plugin contract and the two capability-provider shapes.

Testing & utilities

From @hopdrive/eventkit/testing and the root.

  • fakeSource(opts?) — minimal in-memory source; exposes the raw payload on both contexts.
  • defineFakeEvent(name, detector, jobs, opts?) — assemble a module for tests without a real source.
  • buildDetectorContextFor(...) / buildHandlerContextFor(...) — exercise a detector/job against any source’s context.
  • serializeError / serializeOutput / replaceCircularReferences — the safe-serialization utilities the runtime uses.
  • asEventName / asJobName / asCorrelationId … — branded-id constructors.
