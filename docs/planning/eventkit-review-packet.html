Part 1

v0.1 → v0.3.7 diff

Two phases. Phase A (groups A–E + recovered, v0.2) closed the seams to the existing system. Phase B (v0.3.x refinements) sharpened the plugin/registration model and source naming. Tags: Added = new surface; Changed = behavior/spec altered; Defined = referenced but unspecified, now pinned; Superseded = a v0.2 form later replaced (annotated inline). v0.1 narrative, principles, and ADR-001…010 stay intact.

A

Job data channel

REVIEW #1/#2 · ADR-011
  • Changed§9.4job(fn, options) keeps two args, but options now splits into two typed channels: input (request-scoped, by-reference, never persisted) and metadata (serializable, persisted/recorded). Replaces the overloaded [key]:any bag.
  • Changed§9.4Job function signature becomes (ctx) => …; reads live deps from ctx.input, persisted annotations from ctx.job.metadata.
  • Added§3.8New design principle: persisted and live data are separate channels — the through-line behind durability working at all.
B

Source handler context

REVIEW #4 · ADR-012
  • Added§7Source adapters may implement buildHandlerContext() — a typed contract to enrich the handler context with source data (never detection helpers).
  • Added§9.2HasuraHandlerContext exposes operation, oldRow, newRow, row, plus session-derived role/userId/receivedAt. Restores the escape hatch ADR-007 over-rotated away.
C

Per-invocation entry point

REVIEW #9 · ADR-013 · ADR-016
  • Defined§9.7createEventKit() runs once at module scope and returns an EventKit with handle(payload, request). Replaces listenTo() and per-invocation plugin registration (fixing the warm-lambda leak).
  • Defined§9.7InvocationContext (referenced but undefined in v0.1), plus RequestContext and InvocationResult.
  • Changed§11A non-mutating per-request config hook replaces the deprecated onPreConfigure options-mutation hook. [v0.3.3: renamed onConfigureInvocation → configureInvocation] — it returns a value, so it’s a verb, not an on… notification.
  • Changed§13Loop prevention is now a specified control mechanism (inbound updated_by → envelope.meta.sourceTrackingToken; outbound deterministic ctx.trackingToken stamped into writes), not just an observability field.
D

run() defaults

EVAL LOST-3 · ADR-014
  • Changed§9.5 / §10mode pinned to parallel, continueOnFailure to true — matching today’s Promise.allSettled fan-out. Series/stop-on-failure is opt-in and explicitly does not preserve current behavior.
  • Changed§9.5[Reversed in v0.3 / ADR-018] — v0.2 said run() tolerates falsy entries to preserve cond && job(…). Now the opposite: jobs is a strict JobDefinition[] and a conditional entry is a compile/runtime error. Conditions move to the detector (a named event) or inside the job.
E

Durable execution contract

EVAL LOST-7 · ADR-015
  • Defined§12The BatchJobStatus enum, a hook→status lifecycle table, and the consumer/producer split (matching the real AR split). Job code stops calling batchJob(…). [v0.3.1 superseded the v0.2 API: DurableJobOptions/batchJobs.record()/ctx.batch.record are GONE] — see Phase B below.
+

Recovered planning concepts

EVAL LOST-2/6/7/8/9/10
  • Added§8manuallyInvoked() Hasura helper — so case 'MANUAL': return false suppression of console edits survives the port.
  • Defined§9.6SerializedError + core serializeError/serializeOutput/replaceCircularReferences (promoted from batchjobs).
  • Added§9.3Tiered loggers — detectors get debug only; handlers/jobs get full levels. onLog breadth (incl. timeouts) preserved for Grafana.
  • Defined§9.4JobProgress/JobCheckpoint shapes; progress(value) takes a 0–1 fraction.
  • Added§18EventModule.metadata (description, tags, owner, flow hints, deprecated).
  • Changed§13Observability buffer-and-flush is now MUST, not SHOULD; serverless flush() stated as best-effort.
  • Added§8 / §17 / §14Event-name backward-compat policy; subpath-export bundling CI gate; Compare-Mode “prove one flow first” phasing caveat.
  • Added§22ADRs 011–016 record the why behind each Phase-A amendment.
B

v0.3.x refinements

CHG-7…13 · ADR-017…023
  • Changed§12Durability is emergent from registering BatchJobs — no durable flag. The plugin (registered only in db-batchjobs, requires: ['source:hasura']) injects the batch_jobs row’s input as the job’s baseline ctx.input and self-correlates the row; jobs stay batch-unaware. Configurable periodic log flush feeds the live React view. CHG-7 · ADR-015 rev / 017 / 020
  • Defined§9.8Platform adapters — lambdaPlatform() / netlifyPlatform() (+ planned netlifyV2Platform / netlifyBackgroundPlatform). Time budget collapses to three strategies (native countdown / computed deadline / none); kit.handler() entry. Keeps EventKit runtime-independent. CHG-8 · ADR-021
  • Added§11.1Plugin composition model — three hook shapes (notification on… / delta transform / singleton capability); the built-in default always runs, “default + extra” is native, replacement is DI-injected base not inheritance. Adds augmentEnvelope; tracking-token & order-enrichment stop mutating the payload. CHG-9 · ADR-022
  • Changed§11.4Registration is kit.use(plugin, config?) — pass the plugin/factory itself (not a call); the kit instantiates it. Platform registers the same way (kit.use(netlifyPlatform)); factories renamed *Platform. Naming rule: on… = void notification, bare verb = returns a value. CHG-10 · ADR-019
  • Changed§9.7No object wrapper — the required source is createEventKit’s first positional arg, same (plugin, config?) shape as use(): createEventKit(hasuraEvent) / createEventKit(webhook, { verify }). CHG-11 · ADR-019
  • Added§7Two Hasura source adapters named by transport — hasuraEvent (DB triggers) and hasuraCron (Hasura scheduled triggers, a sibling payload), replacing “hasura + a generic cron.” CHG-12 · ADR-023
  • Changed§3.2Detector preferred style restored to the switch (ctx.operation) house style — named booleans inside each case, sentence return, case 'MANUAL': return false; not the flat insertedReady || becameReady form. CHG-13
Part 2

Decisions

Each is a call only a human can make. Anything left unanswered inherits its Recommendation at ratification. Full text lives in EventKit-open-decisions-202606281830.md.

Resolved (v0.3.x): the five former blocking calls — D1 input/metadata (ADR-011), D2 durable model (now registration-emergent — ADR-015 rev/017/020), D3 run() defaults (ADR-014), D4 entry point (ADR-013), D5 loop prevention (ADR-016) — plus D14 one source per kit (ADR-019). New decisions made along the way: ADR-017 (jobs plugin-agnostic), 018 (declarative handlers), 019 (register API), 020 (augmentJobContext), 021 (platform adapters), 022 (composition model), 023 (hasuraEvent/hasuraCron).
Still open: D6–D13, D15–D18 (below, unchanged), plus four new ones surfaced during the refinement — D19–D22 (bottom). The cards below keep the original context; the chips show current status.

Async detectors

Medium

Allow-but-discourage async (DB-touching) detectors, forbid them, or time-box detection?

CONV risk notev0.2 §8
