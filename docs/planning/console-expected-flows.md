# Planning: EventKit Console (Expected Flows + Compare Mode)

**Status:** planning. The Console itself is still phased (§16, D9). I'm writing the plan
down so nobody has to re-derive it later. It leans on two things that already shipped: the
flow generator (ADR-032) and the Observability records. Everything else here is what we
still have to build.

---

## 1. Where this sits

EventKit answers three questions for every inbound signal. We now have an artifact for each:

| Half | Question | Artifact | Status |
|---|---|---|---|
| **Expected** | What is *supposed* to happen? | the generated flow doc / `toFlowGraph()` (`FlowNode`/`FlowEdge`), ADR-032 | **shipped** |
| **Observed** | What *did* happen? | the Observability Invocation→Event→Job records (correlation-linked, incl. cross-vendor via ADR-028) | **shipped** |
| **Compare** | Where do they *differ*? | the matcher overlaying Observed on Expected, classified | **planned (this doc)** |

The generator (`kit.describe()` and `@hopdrive/eventkit/flow`) already emits the Expected
graph, in the exact `FlowNode`/`FlowEdge` shape the Console needs. The Observability plugin
already emits the Observed records. So Compare Mode is really just the join between those
two, plus a UI. Nothing in the runtime blocks it. What's missing is a backend read layer
and the Console modes themselves.

## 2. What the Console becomes

Today's observability viewer is React Flow over invocation records. It grows into an
architecture explorer with three modes on the *same* canvas:

- **Observed** (ships first, since it's useful before any manifest exists). Render the
  recorded invocation, event, and job tree for a correlation id or invocation, exactly like
  today. The only change is that it reuses the shared node-id scheme (§4).
- **Expected.** Render a flow's `{ nodes, edges }`. That's either the generated graph
  (`toFlowGraph`, structural) or a hand-authored Flow Manifest (business intent, §15). Same
  node kinds (`source | event | job | sideEffect | terminal`), laid out as the canonical
  shape of the process.
- **Compare.** Overlay a specific observed run onto the expected flow, and color each node
  by the matcher's classification. It's never a production gate (§14). It's a lens, not a
  check.

## 3. The matcher (from §14, carried, not re-decided)

Classifications: `expected_missing`, `optional_not_taken`, `condition_not_met`,
`observed_success`, `observed_failed`, `unexpected_observed`, `retrying`, `timed_out`,
`cancelled`, `out_of_order`, `extra_invocation_chain`.

Matcher priority: explicit `expectedNodeId` → `flowId`+name → event name → job name →
source+stage → inferred → unmatched.

The matcher **must preserve uncertainty** (`matchConfidence: 'inferred'`) and **never hide**
an unmatched observed node. An `unexpected_observed` node is a finding, not noise.

Run it on **one** high-value flow first (mobile-service-dispatch, or the Uber-ride flow in
`external-correlation-chaining.md`) before generalizing (§14 phasing caveat, D9).

## 4. Stable node identity (the linchpin)

Observed nodes, expected nodes, generated-graph nodes, and React Flow ids all **have to
agree**, or the overlay is just guesswork. The rule from §14 is simple: **never build ids
from file paths** (they move around in refactors). Use event and job names instead. The
shipped generator already does this:

```
source                              # the one source node
event:<eventName>                   # e.g. event:rideshare.requested
job:<eventName>:<jobName>           # e.g. job:rideshare.requested:callVendorForRide
sideEffect:<jobName>:<effect>       # from job metadata.sideEffect
```

The Observability records already carry `event_name`, `job_name`, `correlation_id`, and
`source_job_id`. That's enough to rebuild the same ids at compare time. **Action item:**
make the observed-to-node mapping reuse these exact id builders. Pull them into a shared
helper in `@hopdrive/eventkit/flow` so both sides import one function.

## 5. Backend (the real unbuilt dependency)

The Console needs a query layer over the Observability storage. The v0.1 API shape still
holds, but we have to nail down the host, auth, and storage queries before it ships (§16):

- `GET /flows/:flowId`. The expected graph (generated + manifest-merged).
- `GET /observations/:invocationId`. The observed tree for one run, plus its correlation chain.
- `GET /compare/:flowId/:invocationId`. Expected overlaid on observed, classified.

Where do the generated graphs come from at request time? A repo commits its `*.flow.yaml`
(generated by `eventkit-flow generate`, gated by `eventkit-flow check` in CI), and the
backend just reads those committed files. No runtime introspection on the hot path. The
observed side is a query over the existing observability tables, which `graphqlSink` already
populates.

## 6. Flow Manifests (the meaning half, still hand-authored)

The generator proves **structure**. A manifest owns **meaning** (§15): the stable flow id,
ownership, required-vs-optional branches, human-readable conditions, terminal conditions,
and cross-event chains (a job writes a row, that triggers another kit's event, and you can't
infer that from one kit). CI validates a manifest against the generated graph (schema,
duplicate ids, edge endpoints, node kinds, terminal reachability, code-reference validation,
generated-graph comparison). `toFlowGraph()` emits the manifest vocabulary on purpose, so a
generated graph can be diffed against a manifest, or promoted into one.

## 7. Build order

1. **Observed mode** on the shared node-id scheme. This is mostly a refactor of today's
   viewer.
2. **Backend read API.** Specify host, auth, and storage queries. Serve the committed
   `*.flow.yaml` as the Expected graph, and observability queries as Observed.
3. **Expected mode.** Render `toFlowGraph` output or a manifest.
4. **Compare mode and the matcher.** Build it on **one** proven flow, then generalize.
5. **Flow Manifest authoring and CI validation.** Promote the generated skeleton into a real
   intent-bearing manifest for that one flow, then add the CI validators.

## 8. Open questions

- **D-console-1:** Where does the matcher live? The Console backend, or a shared
  `@hopdrive/eventkit/flow` function that CI reuses? (Leaning toward a pure function in
  `flow`, so CI and the Console share one implementation.)
- **D-console-2:** Where do manifests live? Per-repo `architecture/*.flow.yaml`, or a
  central registry the backend reads? (Leaning toward per-repo committed, with the backend
  aggregating.)
- **D-console-3:** Backend host and auth. Reuse the existing observability service, or stand
  up a new one? (This blocks §16, and it's still open in D9.)
