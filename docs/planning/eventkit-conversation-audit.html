Section 01

Concepts Captured in RFC

These ideas from the conversation are clearly present in the RFC. Noted briefly — no gaps to fill here.

01
Three-layer model: Expected Flow / Observed Flow / Compare Mode
Captured

The conversation's central insight at line 1504–1515 — three named layers with the color-coded UI semantics (green/gray/yellow/red/purple) — made it fully into RFC §14 and §16. The guiding principle "Expected Flow is the contract. Observed Flow is the evidence. Compare Mode is the truth-finding layer" appears verbatim in both (conversation ~line 2474, RFC §14).

02
Flow Manifests as source-controlled YAML contracts
Captured

The YAML schema (id, name, source, nodes with id/kind/eventName/jobName/required/condition, edges) was sketched at line 1534–1568 and refined in the message-68 design proposal at line 1860–1902. RFC §15 reproduces this faithfully including the TypeScript type shapes for FlowManifest, FlowNode, FlowEdge.

03
Invocation → Event → Job observability hierarchy
Captured

Described in the conversation at line 1496 ("invocation → event executions → job executions → logs") and fully catalogued in the message-68 design proposal at lines 2038–2079. RFC §13 reproduces the three record types with their field lists.

04
Runtime matching metadata: flowId, expectedNodeId, matchConfidence
Captured

Defined at conversation lines 2088–2093 with the exact | inferred | unmatched enum. RFC §13 reproduces this exactly.

05
Matcher priority order (7 levels)
Captured

Priority order at conversation lines 2209–2215. RFC §14 reproduces the same seven-level priority list word-for-word.

06
Compare Mode classification vocabulary
Captured

Classifications defined in the design proposal (lines 2221–2295) including expected_missing, optional_not_taken, condition_not_met, unexpected_observed, out_of_order, extra_invocation_chain. RFC §14 reproduces all. retrying, timed_out, cancelled added in RFC §16 node-states list.

07
Machine-readable console APIs
Captured

Defined at conversation lines 2452–2454 (GET /flows/:flowId, GET /observations/:invocationId, GET /compare/:flowId/:invocationId). RFC §16 reproduces these endpoints exactly.

08
CI validation with three strictness levels and triggered paths
Captured

Conversation lines 2298–2382 define the warn/strict/release levels, the trigger paths, and the generated artifact list. RFC §21 and §8 of the conversation design proposal reproduce this faithfully.

Section 02

Concepts Discussed but Lost or Under-Specified in RFC

These were discussed with meaningful detail in the conversation and did not fully survive the translation to the RFC. Quoted with approximate line numbers from the source file.

01
ObservabilityPlugin transport: 'graphql' — the existing implementation detail and its performance concern
Critical

At conversation line 1496, the assistant notes the existing observability plugin "already tracks invocations, detected events, handlers, job executions, correlation IDs, durations, errors, and job results" and at line 1579 confirms the data model is already working with source_job_id for chained flows. The conversation references actual filecite links to the existing ObservabilityPlugin code — meaning the existing plugin already uses transport: 'graphql' and makes synchronous per-job GraphQL writes.

conversation ~line 1496, 1579
"Its data model already mirrors the runtime execution chain: invocation → event executions → job executions → logs." "Your data model already has the right relationship for chained flows: a job execution can trigger another invocation via source_job_id."

The RFC (§13) specifies what the observability plugin SHOULD capture, but says nothing about the current transport mechanism, whether synchronous GraphQL writes on the hot path are acceptable, or what the migration path looks like from the existing implementation. The field transport: 'graphql' and its hot-path write cost are completely absent from the RFC.

02
Buffering / batching vs. synchronous writes — the explicit performance mitigation discussed
Critical

The conversation design proposal at lines 2086–2096 states "Runtime execution must not fail because no Flow Manifest exists" and the message-68 design proposal for the observability plugin recommends the plugin operate as a lifecycle observer that buffers and flushes. The conversation at line 2028 specifies "Observability should capture runtime execution using the EventKit lifecycle model" — not: "Observability should block on synchronous writes per job."

conversation ~line 1592, RFC §11
"Plugins SHOULD avoid long blocking work inside hot lifecycle hooks. If a plugin needs to persist large payloads, send network requests, or perform expensive computation, it should batch, buffer, or defer that work where possible and flush at invocation end or shutdown."

The RFC's §11 does include this guidance for plugins in general. But the RFC never explicitly calls out the observability plugin's existing synchronous-GraphQL pattern as the specific concern this guidance is meant to address. An implementer reading §13 alone has no visibility that the current implementation does synchronous per-job GraphQL writes and that buffering is the required mitigation. The RFC should name this tension explicitly.

03
Observability plugin failure must not block business execution — stated only in the non-goals, not in the plugin spec
Critical

The conversation design proposal at line 2296 is unambiguous: "EventKit should not block runtime execution because of flow differences. These are observability and validation concerns." The same idea appears earlier in the assistant's message about plugin failures at conversation line 563: "A plugin failure should be recorded and surfaced, but the framework should define which hooks are allowed to fail the invocation and which hooks are best-effort."

conversation ~line 563, design proposal ~line 2296
"Observability write failures, for example, should usually not prevent business execution unless the application explicitly configures strict behavior."

The RFC's §2.3 non-goals mentions this for flow comparison, but §11 (Plugin Architecture) and §13 (Observability Plugin) never state that the observability plugin is classified as best-effort, that write failures should be swallowed and logged rather than re-thrown, or that there is a configurable strict mode. An implementer building the plugin hooks has no normative guidance on exception handling behavior. This is one of the most operationally dangerous gaps.

04
Loop prevention / tracking tokens / detecting the system's own writes
Critical

The existing hasura-event-detector has a tracking token / updated_by provenance mechanism to prevent the system from reacting to its own database writes and entering infinite loops. The conversation references this indirectly through the filecite links to the existing implementation and via the field tracking token appearing in both the invocation record spec and the machine-readable API surface at line 2044 and in the conversation's invocation field list.

conversation design proposal ~line 2044
Invocation Records should include: — tracking token — source job ID if invocation was triggered by a previous job

The RFC §13 lists tracking token as a field to capture in invocation records. But neither the RFC's observability section, source adapter section, nor the migration section ever addresses how loop prevention actually works: how the tracking token is propagated when a job writes to the DB, how the Hasura adapter reads and checks it, and how the system avoids reacting to its own writes. This is a critical operational correctness concern that the conversation surfaced through the source code references but the RFC reduced to a single field name.

05
How chained invocations are correlated — source_job_id threading end-to-end
Notable

The conversation at line 1579 makes a specific point about cross-invocation correlation using source_job_id: a job triggers another DB write → Hasura fires a new invocation → that invocation records its source_job_id linking back to the originating job execution. This makes cross-invocation cascade graphs possible in the viewer.

conversation ~line 1579
"Your data model already has the right relationship for chained flows: a job execution can trigger another invocation via source_job_id. That is important because it lets the viewer show cross-invocation cascades, not just one handler call."

The RFC §13 lists source job ID when an invocation was triggered by a previous job in the invocation record fields. But nowhere in the RFC is the threading mechanism explained: who writes the source_job_id, when (before or after the job executes), whether it's injected automatically by the runtime or manually set by job code, or how the console stitches chains together for rendering. The conversation treated this as a load-bearing design detail; the RFC treats it as a field name.

06
Generated flow manifests vs. hand-authored — the hybrid model and CI enforcement
Notable

The conversation at lines 1582–1603 explicitly distinguishes generated structure (facts that can be derived from code: Hasura trigger name, handler path, event module names, job function names, package imports) from declared manifest content (business meaning: required/optional status, conditions, terminal criteria, ownership). The rule articulated at line 1939: "The manifest describes meaning. The generator verifies structure."

conversation ~line 1582
"Use generated structure for facts: — Hasura trigger name, watched table, trigger operation, handler/function path, event module names, job function names, package imports Use declared flow manifests for business meaning: — this is the 'mobile service dispatch' flow, these branches are optional, — this job is expected only when customer notifications are enabled, — this path terminates when status is dispatched, this failure is retryable, this job is safety critical"

The RFC §15 carries the "manifest describes meaning, generator verifies structure" rule. But it never describes how the generator works: what tool runs it, what inputs it consumes (Hasura metadata YAML? event module exports? package.json graphs?), what outputs it produces, or when it runs in CI. The distinction exists in principle in the RFC but the generation pipeline is entirely unspecified.

07
Redaction / PII / large-payload handling — granularity of decisions
Notable

The conversation's design proposal (message 68, around lines 2038–2050) specifies that invocation records include a source payload reference (not the full payload), implying a reference-vs-inline decision. The RFC §13 mentions this: "The plugin SHOULD support payload redaction and payload references. Full raw payloads may contain sensitive data or large objects."

conversation design proposal ~line 2044
"source payload reference" (not "source payload")

But the conversation's earlier context (the Airbnb rule at line 728: "If a diagram cannot be regenerated, it is untrusted") plus the Stripe-style event-as-product framing implicitly raised the question: should every observability record be queryable without ever needing to re-fetch the original payload? The RFC never specifies what is stored inline vs. by reference, what the reference format is (S3 key? DB row ID? content hash?), or how the console renders redacted fields. For an observability plugin dealing with GDPR-sensitive carrier data, this needs more than a SHOULD.

08
Stable node IDs: file paths explicitly rejected as identifiers
Notable

The conversation design proposal at lines 2199–2205 explicitly called out file paths as unstable:

conversation design proposal ~line 2199
"Do not use file paths alone as stable node IDs. File paths can change during refactors. Prefer semantic names such as event name, job name, source trigger, or exported metadata."

The RFC §15 carries the stable-ID node naming convention (node:<flowId>:<kind>:<semanticName>) but does not include the anti-pattern warning about file paths. This is the kind of nuance an implementer will discover painfully after the first refactor breaks all observability links.

09
Optional metadata hints on event modules and job exports for improved matching
Minor

The conversation design proposal at lines 2005–2023 described optional metadata exports from event modules and jobs to improve matcher accuracy:

conversation design proposal ~lines 2007–2023
export const metadata = { flowHints: ["mobile-service-dispatch"], eventName: "mobility.run.dispatch.requested", }; export const jobMetadata = { name: "sendDriverPushNotification", sideEffect: "push-notification", retryable: true, };

The RFC §18 mentions metadata in the EventModule registration config as an optional field that "may include description, tags, flow hints, owner, deprecated status, and related documentation." But the specific pattern of exporting metadata from within the event module file (so the generator can introspect it statically) is not preserved. The RFC's framing is registration-time metadata, not module-level export metadata. These are different patterns with different static-analysis implications.

Section 03

Decisions Made Differently in Conversation vs RFC

01
Confidence in the matcher: "buildable" in conversation, assumed solved in RFC
Notable

The conversation treated the matcher as something to be proven incrementally. At lines 1642–1652, the recommended path was: "Add flowId and expectedNodeId concepts. Create one expected flow manifest for a high-value flow. Add a matcher that maps observability records onto expected nodes. Do not try to model the whole platform first."

conversation ~line 1652
"Do not try to model the whole platform first. Pick one flow and prove the loop."

The RFC presents the matcher's seven-level priority order, the full classification taxonomy, and the summary format as settled design. The conversation's acknowledged uncertainty about whether this is buildable in practice is completely absent from the RFC. The RFC reads as if Compare Mode is already solved, whereas the conversation treated it as a hypothesis to validate.

02
Flow Manifests: SHOULD in RFC vs. clearly core-path in conversation
Notable

In the conversation at lines 1494–1496, flow manifests were positioned as the natural next step from what's already built: "So the natural next step is not a separate tool. It is: Expected vs actual event flow viewer." The conversation treated this as the primary product direction after the basic observability plumbing works.

RFC §2.1
EventKit SHOULD support Flow Manifests as architecture-as-code artifacts. EventKit SHOULD provide a Console capable of viewing Expected Flow, Observed Flow, and Compare Mode.

The RFC downgrades both to SHOULD under Secondary Goals (§2.1), making them optional. The conversation positioned them as the point of the whole effort — the architectural insight that differentiates this from plain observability. The softening to SHOULD may be a pragmatic RFC choice, but it could lead future readers to deprioritize what the conversation considered the killer feature.

03
Console's role: "interactive architecture explorer" vs. "observability viewer evolving"
Notable

In the conversation at line 1783: "Rather than replacing it, the console should become the interactive architecture explorer for the entire platform." The conversation frames the console as an architecture product, not an engineering debug tool.

conversation ~line 1783
"The existing React Flow visualization in the Observability console is already well suited for this evolution. Rather than replacing it, the console should become the interactive architecture explorer for the entire platform."

The RFC §16 says the Console "should evolve from an observability viewer into an architecture explorer." That's the right direction but the phrasing "evolve from" subtly implies a migration rather than a re-framing of the product's identity. The conversation was making a bolder claim: the console is the architecture platform, not a debug tool that gains some architecture features.

04
Observability plugin storage schema: conversation implies Hasura/GraphQL tables, RFC is storage-agnostic
Minor

The conversation references the existing plugin via filecite links to the actual implementation (using transport: 'graphql'). When the conversation says "observability data is already working" it means it's working against Hasura-backed tables via GraphQL mutations. The conversation design proposal specifies the invocation/event/job record fields assuming they map to database tables.

RFC §13
"The plugin SHOULD provide a stable storage schema that supports the Console, CLI tools, tests, and AI context APIs. Runtime trace data should be queryable by invocation ID, correlation ID, event name..."

The RFC deliberately abstracts away the storage backend (the word "Hasura" does not appear in §13), which is architecturally correct for a generic framework. But this means the existing GraphQL-transport implementation is now architecturally positioned as "a storage adapter choice" — without any guidance on how to implement that adapter interface, what the schema migration looks like, or whether the existing Hasura tables are the reference implementation.

05
Business flow graph as "the most valuable graph" — framing lost in RFC
Minor

The conversation at lines 1386–1388 made a strong prioritization claim that the business event/flow graph is more valuable than the package dependency graph for HopDrive specifically. This framing motivated why Compare Mode matters.

conversation ~line 1386
"The most valuable graph for HopDrive is probably not the dependency graph. It's the business flow graph."

The RFC describes all the pieces but does not reproduce this prioritization reasoning. A future reader of the RFC will not understand why flow manifests receive so much design attention relative to, say, the package graph or the data graph. The motivating argument — that change impact in an event-driven logistics platform is primarily a business-event tracing problem, not a code-import problem — is not in the RFC.

Section 04

Explicitly Rejected Ideas

01
Hand-maintained architecture prose / wiki diagrams
Rejected

Rejected at lines 499–501 and reinforced by the Airbnb rule at line 728.

conversation ~line 499, 728
"The unsafe version is a separate hand-written diagram that says one thing while the code does another. The safe version is a generated graph with annotations." "If a diagram cannot be regenerated, it is untrusted." — Airbnb's rule, cited approvingly

This is clearly reflected in the RFC's CI validation and generated artifact requirements. The rejection is preserved.

02
Compare Mode as a production execution gate
Rejected

Explicitly rejected at conversation design proposal line 2296: "EventKit should not block runtime execution because of flow differences. These are observability and validation concerns." Preserved in RFC §2.3 Non-Goals and §14.

conversation design proposal ~line 2296
"EventKit should not block runtime execution because of flow differences. These are observability and validation concerns. CI and tests may enforce strictness separately."
03
Nx as primary build system (for current HopDrive scale)
Rejected

Explicitly rejected at lines 1104–1117 in favor of Turborepo + Dependency Cruiser. The conversation notes Nx brings "more framework, more magic, more configuration, higher onboarding burden" and that agents "frequently need Nx-specific context."

conversation ~line 1104
"I would not start with Nx. If HopDrive reaches 50+ apps/packages/functions, then reevaluate. Right now Turborepo gives 80% of the value for 20% of the complexity."

This rejection is not in the RFC at all (the RFC is scoped to EventKit, not monorepo tooling). But it is worth noting this decision was made definitively.

04
Backstage as Phase 1 tooling
Rejected

Backstage was evaluated at lines 1230–1303 and deferred explicitly: "Not first. I'd wait until after monorepo consolidation. Maybe phase 2." The EventKit Console was positioned as the HopDrive-specific replacement for Backstage's service-catalog and architecture-explorer functions.

conversation ~line 1299
"Not first. I'd wait until after monorepo consolidation. Maybe phase 2."

This rejection provides important strategic context for the Console's ambitions in §16 — the Console is intentionally filling the gap Backstage would otherwise fill, at lower operational cost. The RFC does not explain this positioning.

Section 05

Notable Concerns and Risks the RFC Ignores

01
Synchronous per-job GraphQL writes on the hot execution path
Critical

The existing observability plugin makes synchronous GraphQL mutations for each invocation, event detection, and job execution record. In a Netlify function processing a Hasura trigger, this means every invocation does N+1 serial round-trips to Hasura on the critical path before returning. The conversation implicitly acknowledged this by recommending the plugin model with buffering and flush-at-end.

RFC §11 (general plugin guidance)
"Plugins SHOULD avoid long blocking work inside hot lifecycle hooks... it should batch, buffer, or defer that work where possible and flush at invocation end or shutdown."

The RFC puts this in §11's general plugin guidance but never calls out the observability plugin specifically as the high-risk case. An implementer who doesn't connect these dots will build an observability plugin that adds 200–400ms of GraphQL latency to every production invocation. The RFC needs a normative statement in §13: the observability plugin MUST buffer writes and flush at onInvocationEnd rather than writing synchronously in onJobEnd.

02
Matcher speculative complexity — risk of over-engineering before validation
Critical

The conversation explicitly stated: "Do not try to model the whole platform first. Pick one flow and prove the loop." (line 1652). The seven-level matcher priority, full classification taxonomy, and summary format are in the RFC as if already proven. There is a real risk that Compare Mode as specified is too complex to implement well before the simpler pieces (basic observability, one flow manifest, one comparison) have proven the value.

conversation ~line 1652
"Start narrow: Add flowId and expectedNodeId concepts. Create one expected flow manifest for a high-value flow, probably mobile-service dispatch. Generate React Flow nodes/edges from that manifest. Add a matcher that maps observability records onto expected nodes."

The RFC has no phasing for Compare Mode. It specifies the full design end-state without indicating which parts are MVP vs. future. The conversation's caution about validating the loop first before modeling the whole platform is completely absent.

03
Loop prevention mechanism has no specification
Critical

HopDrive's existing hasura-event-detector has a mechanism (tracking token / updated_by field checking) to prevent an event handler that writes to the DB from triggering itself infinitely. This is a correctness-critical invariant. The RFC carries the tracking token field in the observability record spec but never explains how the mechanism works at the source adapter level.

RFC §13 (invocation record fields)
Invocation records should include: [...] tracking token

Questions left unanswered by the RFC: Does the Hasura source adapter check the tracking token before invoking the runtime? Does the tracking token propagate into job context so jobs can stamp their writes? Is loop prevention the adapter's responsibility or the application's? Without this spec, the migration from hasura-event-detector breaks a correctness property silently.

04
Observability plugin failure behavior: no normative error-handling contract
Notable

The RFC §11 notes plugin failures "must be handled carefully" and that "Observability write failures, for example, should usually not prevent business execution." "Should usually not" is not normative enough for a safety-critical logistics platform where a flaky Hasura connection during observability writes must never drop a legitimate dispatch event.

RFC §11
"Plugin failures must be handled carefully. A plugin failure should be recorded and surfaced, but the framework should define which hooks are allowed to fail the invocation and which hooks are best-effort."

The RFC says the framework "should define" this — but then does not define it. Which hooks for the observability plugin are best-effort? All of them? Only onJobEnd? What about flush() failure — should that fail the invocation? This needs a concrete contract, not a statement that a contract should exist.

05
Flow manifests potentially becoming stale faster than CI catches them
Notable

The conversation's Airbnb rule (line 728) and the CI strictness levels exist precisely because manifests will drift. But the RFC only specifies CI-level enforcement. It does not address: what happens when a job is renamed mid-sprint but the manifest CI check runs only on PR? Local development will show manifest mismatch warnings but the developer may ignore them. The RFC's warn/strict/release levels are defined but there is no guidance on how developers are expected to discover that their local manifest is stale before pushing.

RFC §21
"local development: warn; pull request: strict; release branch: release"

No developer tooling is specified for local manifest staleness discovery — no eventkit flow check IDE integration, no watch-mode validation, no pre-commit hook guidance. The RFC specifies CI levels but not the local feedback loop.

06
The Console requires its own backend — completely unaddressed
Notable

The Console as described in RFC §16 needs to serve GET /flows/:flowId, GET /observations/:invocationId, and GET /compare/:flowId/:invocationId. These are not Netlify function endpoints — they're query APIs over observability tables and generated flow manifests. The conversation assumed the existing React Flow console would evolve, but the existing console reads from Hasura directly via GraphQL.

conversation ~line 1496 (filecite references to existing console)
"The console already renders those runtime nodes and edges with React Flow, using invocations plus correlated invocations as graph input."

The RFC specifies machine-readable API endpoints as if they exist, but provides zero guidance on: where these APIs run (Netlify function? separate service?), how they query the observability storage, or how they merge flow manifest JSON with observability records at query time. This is a significant implementation gap that the conversation's scoping left implicit.
