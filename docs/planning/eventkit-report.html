§1 CONCEPTS CAPTURED IN RFC — confirmed agreed

These concepts were explicitly agreed in the conversation and appear to be reflected in the final RFC. They are listed here for orientation and completeness.

CONFIRMED

Agreed concepts that appear in the RFC

The following were explicitly resolved in the conversation (primarily messages [9]–[16], lines 583–2379) and are confirmed present in the RFC:

  • Source-agnostic runtime chain: EventEnvelope → DetectorContext → DetectedEvent → HandlerContext → run() → JobExecution[]
  • createEventKit({sources, plugins, events}) with explicit module registration — no filesystem auto-discovery
  • job(fn, JobOptions) factory; JobOptions = {name, timeoutMs, retries, tags, metadata, durable, continueOnFailure}
  • run(event, jobs, options) with mode: 'parallel' | 'series'
  • EventKitPlugin lifecycle hooks (full set: onInvocationStart → onJobEnd, etc.)
  • BatchJobs as durability plugin: Core owns retry mechanics; BatchJobs owns persisted retry state/history
  • One detector per event module; detector always first; handler below
  • Single @hopdrive/eventkit package with subpath exports
  • Source-specific detector contexts: HasuraDetectorContext, WebhookDetectorContext, CronDetectorContext
  • Flow Manifests (design-time contract), Expected/Observed/Compare modes, console architecture
  • ADRs, testing strategy, CI validation with strictness levels (warn/strict/release)
  • Compatibility facade for @hopdrive/hasura-event-detector (duration deferred)
§2 CONCEPTS LOST OR UNDER-SPECIFIED IN RFC

These were discussed and resolved in the conversation but appear to be absent or vague in the final RFC. Each entry includes a direct quote with line number.

GAP 1

How a handler passes data into jobs — the input/deps channel

P1 Critical

The conversation made an explicit, firm decision that DetectorContext does NOT survive into handlers or jobs, and that handlers access source data via ctx.envelope.payload (not by re-fetching). This was stated twice:

"Handlers should not use detector-only helpers like columnChanged(). If handlers need source data, they use ctx.envelope.payload or source adapter-provided handler context extensions."

Lines 2063–2065

"Important decision: DetectedEvent should not carry DetectorContext. Detector helpers are for detection only."

Lines 2007–2008

The conversation also defined a HasuraHandlerContext extension that gives handlers typed oldRow / newRow access without the detector helpers:

export interface HasuraHandlerContext<TNewRow, TOldRow = TNewRow>
  extends HandlerContext<HasuraEventPayload<TNewRow>> {
  operation: HasuraOperation;
  oldRow: TOldRow | null;
  newRow: TNewRow | null;
}

Why it matters: The current hasura-event-detector passes business data in the job options bag. The RFC defines JobContext with event.envelope.payload but does not explicitly state that jobs consume the raw Hasura payload directly (no intermediary typed input). If unspecified, implementers may invent their own data-passing convention — exactly the inconsistency the RFC should prevent.

GAP 2

Execution mode default and failure isolation — no default stated

P1 Critical

The conversation defined RunOptions with mode?: 'parallel' | 'series' but never stated a default value. The current library uses Promise.allSettled (parallel, failures do not stop other jobs). This is load-bearing behavior.

export interface RunOptions {
  mode?: 'parallel' | 'series';
  continueOnFailure?: boolean;
  timeoutMs?: number;
  metadata?: Record<string, unknown>;
}

An earlier handler example (line 773) showed Promise.all usage, but that was user-authored code, not a framework default specification.

"The framework shouldn't prescribe orchestration beyond providing run()."

Line 778

Why it matters: If the RFC's default is series or if continueOnFailure defaults to false, migrating consumers will silently change behavior. The conversation never locked this in. The RFC must state the default explicitly — and the current library's Promise.allSettled behavior (parallel + isolated failures) is the most natural default to match.

GAP 3

Tracking token — mentioned in observability data but dropped from public types

P1 Critical

The observability section listed trackingToken as a first-class invocation correlation concept alongside correlationId and traceId:

"Invocation Records: invocation ID, source adapter kind, source function, correlation ID, trace ID, tracking token, source payload reference..."

Lines 1068–1078

"Load observed invocations by invocation ID, correlation ID, trace ID, or tracking token."

Line 1155

But the final agreed EventEnvelope type (lines 1968–1981) has: id, source, sourceType, receivedAt, correlationId, payload, meta, raw? — no trackingToken. The concept was captured in the observability data model but never wired into the framework's public types or specified as to its origin (header? generated? caller-supplied?).

Why it matters: The current library uses x-hasura-client-name for loop-prevention and updated_by provenance. trackingToken appears to be the EventKit successor to that mechanism, but it was not resolved where it lives in the type hierarchy.

GAP 4

Loop prevention / updated_by provenance — noted in observability, not in framework types

P1 Critical

The observability data model included a critical anti-loop field:

"source job ID if invocation was triggered by a previous job"

Line 1079

This field implies the framework can detect when a Hasura event was generated by a prior job's DB write — the core mechanism for preventing event storms. But neither the EventEnvelope type, the DetectorContext, nor any plugin lifecycle hook was specified as the mechanism for surfacing this at runtime. It appears only as an observability data field, not as a framework control primitive.

Why it matters: Event storms are a real operational risk in any event-driven system. The conversation captured the concern in the data model but left the enforcement mechanism completely unspecified. The RFC must answer: who populates sourceJobId? How do detectors or the framework use it to suppress loops?

GAP 5

Package import convention — code examples are inconsistent throughout

P2 Medium

Three competing import forms appear in the conversation, even after the agreed form was established:

FormAppearanceStatus
'@hopdrive/eventkit-source-hasura'Lines 1776, 1800, 2299Pre-decision; rejected at line 851
'@hopdrive/eventkit/sources/hasura'Lines 851, 1894Agreed form

GPT acknowledged this at lines 1891–1903 and recommended fixing all examples to the subpath form, but did not do so. The BatchJobs end-to-end example (lines 2297–2318) still uses the multi-package form.

Why it matters: The RFC must normalize all code examples to the agreed subpath imports, otherwise implementers will copy the wrong form.

GAP 6

run() signature — two different call shapes in the conversation

P1 Critical

Two distinct run() shapes appear:

Shape A — standalone import (final agreed type, lines 2156–2163):

export async function run<TPayload, TMeta>(
  event: DetectedEvent<TPayload, TMeta>,
  jobs: JobDefinition[],
  options?: RunOptions,
): Promise<JobExecution[]>;

Shape B — context-injected (earlier example, line 760):

export const handler = async ({ run }) => {
  await run(sendNotificationJob);
  await run(updateAnalyticsJob);
};

In Shape B, run is destructured from the handler context and called with a single job — a fundamentally different API. The conversation ended with Shape A but Shape B appeared in discussion and was never explicitly dismissed.

Why it matters: The handler's public API shape determines the entire developer experience for every event module author. This must be unambiguous in the RFC.

GAP 7

Async detectors — performance concern never discussed

P2 Medium

The agreed DetectorFunction type allows async detectors:

export type DetectorFunction<TContext extends DetectorContext> =
  (ctx: TContext) => boolean | Promise<boolean>;

This means detectors can make DB or API calls. In a system with 50–100+ event modules, an async detector that hits the DB runs for every invocation against every registered module. The conversation never discussed:

  • Whether async detectors are allowed in practice or just in type
  • Whether they run concurrently or serially per invocation
  • Latency implications of N-module × M-DB-call evaluation
  • Timeout/cancellation for async detectors

Why it matters: Latency amplification from naive async detector evaluation could make EventKit unacceptable in production without prior guidance. The RFC should state the expected usage convention even if the type allows async.

GAP 8

Detection ordering — first-match vs all-match never resolved

P2 Medium

The conversation never specified whether all registered detectors are evaluated for every invocation, or whether the framework stops at the first positive match.

This has significant correctness and performance implications:

  • In the current library, all modules are evaluated (all-match)
  • All-match allows multiple business events per invocation — is this intended?
  • First-match would be a behavioral regression from the current library
  • If all-match, do their handlers run concurrently or in registration order?

The RFC must define this guarantee explicitly, since it affects both performance modeling and correctness of event authoring.

GAP 9

Handler function signature — type definition and examples show different shapes

P2 Medium

Agreed type (lines 2216–2221):

export type HandlerFunction<TCtx extends HandlerContext> = (
  event: DetectedEvent,
  ctx: TCtx,
) => Promise<JobExecution[]> | JobExecution[];

Example code throughout (e.g., line 2224):

export const handler = async (event, ctx) => {
  return run(event, [
    job(updateMoveTimeline),
    job(notifyDealer),
  ]);
};

The two-argument form (event, ctx) is the agreed type. The single-argument destructuring form ({ run }) appeared earlier and was never explicitly deprecated. The RFC should confirm the two-argument form as canonical and explain that run is a named import, not a handler context property.

GAP 10

DurableJobOptions type — referenced but never defined

P2 Medium

JobOptions.durable is typed as boolean | DurableJobOptions (line 2109), and the BatchJobs example shows:

job(runARBatchV2, {
  durable: batchJobs.record(ctx.newRow),
})

But what batchJobs.record() returns, and what shape DurableJobOptions takes, was never specified in the conversation. Similarly, the progress() and checkpoint() methods on JobContext (lines 2132–2134) were listed without specifying what they persist to, who reads them, or whether this is a Core or BatchJobs feature.

Why it matters: These are the primary BatchJobs integration points. Leaving them undefined means the BatchJobs plugin author must invent the contract, which is exactly what the RFC should prevent.

GAP 11

manuallyInvoked() — helper present, usage convention absent

P3 Low

The agreed HasuraDetectorContext includes manuallyInvoked(): boolean (line 1657), but the conversation never explained its purpose, when to use it, or the convention for detectors that should suppress re-detection on Hasura console re-delivery. In Hasura, "manually invoked" means the event was triggered via the Console "Redeliver" button or the x-hasura-event-type: manual header.

Why it matters: Without documentation, detectors that need idempotency guarantees on re-delivery will not know to check manuallyInvoked(). This is directly related to the loop prevention / event storm concern.

GAP 12

"Why EventKit Exists" strategic framing — added to Google Doc, may be absent from RFC

P2 Medium

The very last substantive exchange in the conversation (lines 5701–5754) established the strategic framing section that Rob explicitly requested:

"EventKit is one of the foundational abstractions that HopDrive intends to build upon for the next decade. It standardizes how business events are discovered, executed, observed, documented, and reasoned about by both humans and AI agents. Its purpose is not simply to replace the Hasura Event Detector, but to establish a stable architectural language that survives changes to infrastructure, event sources, and execution environments."

Lines 5715–5720

GPT added this to the Google Doc in message [134] and also proposed the architectural philosophy:

"Infrastructure detects. Business events describe. Handlers respond. Plugins observe."

Lines 5750–5751

Why it matters: This framing appeared at the END of the conversation, after the RFC chapters were already generated. The RFC Markdown files (RFC-01 through RFC-09) were produced before this framing was added to the Google Doc. It may be missing from the RFC entirely. This is the document's "north star" and should appear near the top.

DIFF 2

Retry ownership — explicit split agreed after a noted contradiction

A contradiction was identified in the source files between two descriptions of retry ownership (lines 501–525). The agreed resolution (lines 668–673):

Core ownsBatchJobs owns
Execution semantics, lifecycle, retry algorithm, retry decisions Persistence, retry scheduling, state transitions, retry history, delayed execution, durability

If the RFC assigns retries differently — e.g., giving all retry logic to BatchJobs — it contradicts the explicit consensus resolution.

DIFF 3

DetectedEvent must NOT carry DetectorContext — explicit decision

An earlier draft of DetectedEvent (in message [13], lines 1852–1868) included detectorContext: TSourceContext. The final agreed type (lines 1987–2005) removed it. The decision was stated explicitly:

"Important decision: DetectedEvent should not carry DetectorContext. Detector helpers are for detection only."

Lines 2007–2008

If the RFC's DetectedEvent interface includes detectorContext or sourceContext, it directly contradicts this consensus. Check the RFC's type definition carefully.

DIFF 4

Flow Manifests are explicitly post-MVP — not a Phase 1 requirement

"Flow Manifests should not block the MVP rewrite. They should be included in the architecture and package design, but likely implemented after the core execution, BatchJobs, and Observability foundations are stable."

Lines 1490–1491

"EventKit should not block runtime execution because of flow differences. These are observability and validation concerns. CI and tests may enforce strictness separately."

Lines 1303–1304

The intended phase ordering: Core → BatchJobs → Observability → Flow Manifests → Compare Mode → Console. If the RFC presents Flow Manifests alongside core execution as co-equal requirements, this phasing decision is lost.

DIFF 5

Single-detector rule — the consequent rule for future sources was stated but may be missing

The reasoning behind one-detector-per-module was explicit:

"If another source can detect the same business event in the future, I'd rather have a second event module than complicate the primary authoring model. We can always revisit this years from now if it becomes a real need."

Lines 688–689

This means: if a business event must be detected from a second source, create a second event module (possibly with a variant name), not a second detector in the existing module. This consequent rule was stated in the conversation but may not appear explicitly in the RFC.

§4 EXPLICITLY REJECTED IDEAS — must appear as ADRs

These alternatives were explicitly considered and rejected during the conversation. They should each have a corresponding ADR in the RFC so future contributors do not re-open settled debates.

REJ 1

Multiple detectors per event module

Reason rejected: Complicates the authoring model. "One detector keeps the authoring experience extremely simple... If another source can detect the same business event, I'd rather have a second event module." (Lines 679–689)

Future path: Revisit only if a real multi-source need emerges; create a second module in the interim.

REJ 2

Filesystem auto-discovery of event modules

Reason rejected: "Trying to discover modules dynamically in serverless environments almost always becomes more complicated than it's worth." Non-deterministic, bundler-hostile, not tree-shakeable. (Lines 748–749)

Chosen alternative: Explicit registration in createEventKit({ events: [...] }).

REJ 3

Separate npm packages per adapter and plugin

Reason rejected: "I'd rather not have a separate package for each plugin and adapter. I'm worried it will clutter the HopDrive scope." Maintenance overhead, namespace pollution. (Lines 656–658)

Chosen alternative: Single @hopdrive/eventkit with subpath exports.

REJ 4

HopDrive domain logic inside source adapters or plugins

Reason rejected: Breaks source-agnosticism. "Framework code may understand Hasura concepts such as: operations, rows, schemas, tables. It must never understand business concepts such as: moves, drivers, appointments, dealerships." (Lines 3499–3508)

Application helpers (e.g., isRideMove(row)) belong in event-handlers/lib/.

REJ 5

Manually maintained architecture diagrams

Reason rejected: "Rather than maintaining architecture diagrams by hand, the system should generate dependency graphs directly from source code, package imports, Hasura metadata, event triggers, Netlify function manifests, GraphQL operations, and SDK definitions." (Lines 153–157)

Diagrams drift from reality; architecture-as-code preferred.

REJ 6

Compatibility facade as a permanent strategy

Duration explicitly deferred: "Not sure but I don't really care about depreciation right now." (Line 658). The facade is a time-limited migration aid, not a long-term commitment. GPT noted: "Compatibility facade duration: Not important right now. Can mark as 'defer.'" (Lines 1506–1507)

§5 MIGRATION / PHASING NUANCES AND RISKS

Issues raised in the conversation about sequencing, scope, and risk that the RFC should reflect accurately.

RISK 1

Compatibility facade scope is unknown — could be trivial or massive

The conversation listed the compatibility facade for @hopdrive/hasura-event-detector as a migration tool but never analyzed what APIs the facade must emulate. The conversation never established how many event modules, job functions, or consumers use the current library's public surface. Without this, the migration phase estimates are speculative.

RISK 2

RFC Markdown files are acknowledged as compressed/incomplete drafts

"I can't generate a true 40 to 60 page, fully realized RFC with hundreds of code examples, diagrams, and API reference in a single response. That exceeds the model's output limits, even if I write it directly to a file."

Lines 3114–3116

"While these chapters accurately capture the architecture and decisions we've made, they're intentionally concise because of generation limits."

Lines 4145–4146

The Google Doc was intended as the canonical living spec; the RFC Markdown files (RFC-01 through RFC-09) were acknowledged as compressed summaries. The "final RFC" being compared may be less complete than what was iteratively added to the Google Doc through messages 66–134. The Google Doc at docs.google.com/document/d/18Qk0-fufGjF-46hPvLR-wF7Iq-QJ5vdyY4-F5_odgtI (line 4371) may be more complete.

RISK 3

ADR count mismatch — 7 named in conversation, RFC lists 10

The conversation explicitly named 6 ADRs (lines 2541–2558) plus one more in message [58] (ADR-007: One Detector Per Event Module, lines 4289–4316) = 7 total. The RFC lists "ADRs 001-010" — three additional ADRs were added during RFC generation and were never reviewed or confirmed with Rob. Those three unreviewed ADRs should be identified and verified.

RISK 4

Phasing: Flow Manifests explicitly deferred to post-Core/BatchJobs/Observability

The intended implementation sequence (from message [12], lines 1490–1491):

  1. Core execution pipeline
  2. BatchJobs plugin
  3. Observability plugin
  4. Flow Manifests + CI validation
  5. Compare Mode
  6. Console architecture explorer

If the RFC presents all phases as co-equal or in a different order, this phasing decision is lost — and Flow Manifests become a gate on an otherwise-ready core rewrite.

§6 NOTABLE CONCERNS THE RFC IGNORES

Issues that were either not discussed or only briefly acknowledged in the conversation, and which appear to be absent from the RFC's surface.

CONCERN 1

Re-fetching data per job — GraphQL round-trips unaddressed

Jobs receive ctx.event.envelope.payload (the raw Hasura snapshot payload). Jobs that need the current database state must re-fetch. The conversation never addressed:

  • Whether jobs are expected to re-fetch or use the snapshot
  • Whether a shared data layer or request cache is recommended
  • Cost/latency of N jobs each making independent GraphQL calls
  • Consistency guarantees between the event snapshot and re-fetched data
CONCERN 2

Blast radius unknown — no count of existing modules, jobs, or consumers

The conversation never established how many event modules, job functions, or consumers currently exist in hasura-event-detector. Without a blast-radius estimate, migration phase timing and compatibility facade scope are unanchored. A single "surprises the team" number here could substantially change the migration plan.

CONCERN 3

Detector readability is "above all else" — enforcement mechanism unresolved

"The framework should optimize for this style above all else."

Lines 111–113

Whether this strong preference is enforced (lint rules, CI checks on detector style) or merely encouraged was never resolved. The RFC should state the enforcement strategy, even if the answer is "convention only for now."

CONCERN 4

Console and Compare Mode are long-term features presented as co-equal with core

The console architecture (Expected/Observed/Compare modes) is explicitly long-term vision. The conversation recommended starting with a single high-value flow (Mobile Service Dispatch) and proving the pattern before scaling. If the RFC presents the console as a Phase 1 requirement alongside core execution, it inflates MVP scope significantly.

CONCERN 5

Monorepo/Turborepo stack conflated with EventKit rewrite

One of the source snippets described a platform-level monorepo consolidation (pnpm workspaces, Turborepo, Dependency Cruiser — lines 144–157). This is a separate, massive project from the EventKit rewrite. EventKit does not require the monorepo. If the RFC conflates these, it creates false dependencies and inflated scope.

CONCERN 6

AbortSignal, progress(), and checkpoint() — in types but never specified

Three JobContext members (lines 2132–2135) — signal?: AbortSignal, progress(), checkpoint() — appear in the agreed types but were never discussed. Cancellation semantics (who signals, what guarantees jobs honor it) and the persistence target for progress/checkpoint events are completely unspecified. This is implementation-detail debt that the RFC leaves to implementers.
