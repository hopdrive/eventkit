Simplification + declarative-response pass: internals consolidated, the authoring surface extended (source-scoped `defineEvent`), and the response seam redesigned to be self-describing (one breaking rename, pre-adoption).

- **Removed dead export `NotImplementedError`** (root and `/core`). It was a Phase-0 scaffolding relic: no runtime code path ever threw it, so nothing could ever catch it. Migration: delete any import of it — there is no replacement because there was no behavior.
- The three Hasura sources (`hasuraEvent`, `hasuraCron`, `hasuraAction`) now share one `callableSource` assembly helper instead of three copy-pasted factory blocks; their runtime shape (callable + attached authoring helpers + plugin `name`) is pinned by new tests.
- `handle()` and `dryRun()` share one intake pipeline (extract → normalize → augment), and the response seams share one error-mapping path — the invocation lifecycle now reads in one place.
- Consolidated six copies of the `crypto.randomUUID` fallback and two copies of `isJobDefinition` into single internal helpers; the two Netlify v2 platforms share the Web-`Response` rejection formatter.
- **`webhook` factory now carries the typed authoring helpers** (additive): `webhook.detector<TBody>(fn)` / `webhook.prepare<TBody>(fn)` work on the bare factory value (uniform with the Hasura family), so an event module can type `ctx.body` without constructing the vendor-configured source that lives in the entry file. New `WebhookAuthoring` interface; runtime behavior unchanged (identity wrappers).
- **Source-scoped `defineEvent`** (additive): every source now carries a typed module builder — `hasuraEvent.defineEvent<Row>({ … })`, `webhook.defineEvent<Body>({ … })`, `hasuraAction.defineEvent<Input>({ … })`, `hasuraCron.defineEvent<Payload>({ … })`. The one type parameter on the outer call types every inline `(ctx) => …` seam (`detector`, `prepare`, the `response` fns) with the source's enriched context, so detectors can be authored inline with no per-seam wrapper. At runtime it is core `defineEvent` (name branding only). New core type `SourceEventModule` (type-only). The generic root `defineEvent` is unchanged and remains the full-inference path (`ctx.prepared` from `prepare`, D32).
- **BREAKING (pre-adoption): `resolve`/`respond` replaced by the declarative `response` field** (ADR-026, amended). A module now declares its reply as one field whose key states the behavioral contract at the definition site: `response: { static: body }` (a constant reply — data, not code, so it provably cannot depend on or wait for the work), `response: { fromRequest: (ctx) => output }` (computed from the request; runs alongside the jobs, structurally cannot see their results; a throw maps to `ClientError`/`ActionError` on the wire), or `response: { fromJobs: (ctx, { jobs, ok }) => output }` (computed from the jobs' outcomes; sequenced after they settle; requires ≥1 job; rejected under 202/background platforms). The single field makes the modes mutually exclusive by construction, and dual keys fail to compile. Migration: `resolve: fn` → `response: { fromRequest: fn }`; `respond: fn` → `response: { fromJobs: fn }`; a constant-returning resolve → `response: { static: body }`. Registering a module with the old fields throws with this mapping in the message. `FlowResponseKind`/`kit.describe()` now report `'static' | 'from-request' | 'from-jobs' | 'none'`, derived from the declaration (never authored, so it cannot drift). The `hasuraAction.resolve`/`webhook.resolve` per-seam helpers are removed with the field; `ResolveFunction`/`RespondFunction` types are replaced by `ResponseDeclaration`/`ResponseBody`.
- **`registerEvent`/`registerEvents` now accept `EventModule<any, any, any>`** (type-level widening): a source-typed module registers without a variance cast under strict compiler flags (e.g. `exactOptionalPropertyTypes`). Runtime unchanged.
- `docs/guide.html`: documented `kit.dryRun()` and the Hasura sources' factory/config form (inbound token discovery) in the API reference; replaced the webhook walkthrough with a real Stripe `payment_intent.succeeded` example (now authored with `webhook.defineEvent<StripePaymentEvent>` and a declared `response: { static }`) and the `hmacVerify` preset (the previous example called `.detector` on the bare factory — which didn't exist before this change — and showed retry semantics that don't match Stripe's redeliver-on-any-non-2xx contract).
